## 一分钟速览

- **谁**：独立安全研究员、数字循真创始人 Lucifiel，自 2025-12 起是 Donut 的用户与 beta 测试者。
- **授权**：Donut 官方两届 Bug Hunt（2026-02 的 1.0、2026-03 的 2.0）及后续线上沟通框架内。
- **发现**：Donut Browser 与 D0 两套系统共 **49 个问题**（Critical 10 / High 19 / Medium 16 / Low 4），核心两条：
    - **① Donut Browser 自动签名盗币链**：后端在用户无签名确认下即可自动签名并广播交易，且可通过诱导点击远程触发（自有账户已实证，11 笔链上交易 Solscan 可查）；跨用户转账请求在平台侧可被构造放行，最终由第三方托管 Turnkey 拦截、未实际转走他人资金。🎬 [脱敏 PoC 演示视频](./assets/poc-demo.mp4)（GitHub 点开即播；亦见[博客内嵌版](https://www.lucifiel.com/donut-disclosure/#poc-demo)）
    - **② D0 远程命令执行**：在研究员自有 Pod 内取得 RCE（已实证）。
- **流程**：经三次线上会议 + 脱敏报告 + PPT + PoC 视频 + 书面通知告知厂商，给出超 90 天修复窗口，产品负责人当场确认披露流程。
- **结果**（截至 2026-06 复测）：
    - **修复状态**：49 项中**仅 11 项完全修复**，9 项部分修复，29 项仍未修复（含多个核心 Critical）。
    - **处理口径**：系统级研究（49 漏洞 / 攻击链 / RCE）被项目方按常规社区奖励口径处理，未获任何单独评估或专项赏金，也未在任何公开渠道（官网致谢、安全名人堂、漏洞披露致谢等）获得正式认可（仅 Discord 社区有非正式感谢）；两届 Bug Hunt 活动各 100 USDC、共 200 USDC，是研究者参赛、按活动规则赢得的常规活动奖金（其中 Bug Hunt 2.0 的 100 USDC 凭独立提交的 13 个漏洞赢得），与本次系统级研究无关；研究者早在 2026-03-18 即书面要求将参赛活动奖金与系统级研究分开评估、反对把后者并入常规社区奖励口径。
    - **最终回应**：研究员发出最终通知后，项目方未就诉求继续沟通，转以法务立场回复、声明保留追责权利，**并作出多条不实指控**（均与会议及书面记录不符，正文已逐条对照；会议原始语音与书面记录均已完整封存、脱敏收录，欢迎机构监督核实）。
- **技术支持**：数字循真平台研判框架（攻击链建模与风险研判）。
- **本文**：完整脱敏、零 PoC、零可复用武器的负责任披露记录。

## 事件背景

本文作者是一名安全研究人员。2025 年 6 月，研究员因 Donut 的项目叙事而关注该产品并加入其官方 Discord 社区；2025 年 12 月参与官方产品访谈与 beta 测试。在 Donut 推出官方安全活动（Bug Hunt）后，研究员转入对其的系统性安全研究，并先后参加两届 Bug Hunt 活动。

在两届活动及后续沟通框架内，研究员对 Donut Browser 与 D0 两套系统累计发现 49 个安全问题，主要涉及钱包交易授权、用户隔离、资产归属校验、控制面访问与运行环境隔离，涵盖可影响用户资金安全的攻击链（11 笔链上交易为证）与 D0 平台的远程命令执行等核心风险。研究员的诉求自始公开、一致：早在首次会议就明确表达想加入 Donut（原话“如果有可能，我也比较想参与到这边来”“很想参与进来”，并提到此前已与[首席运营官]谈过这一想法），早期 Telegram 沟通中又强调“个人更在意的是长期合作”，第三次会议进一步明确希望全职加入团队。

在赏金问题上，研究员始终表示金额可沟通，并多次请项目方按漏洞影响、参照行业标准自行评估报价；直到约三个月后、由项目方主动邀请洽谈的第三次会议，研究员才给出自评区间并说明计价依据，同时重申请项目方参考行业标准定价。全程定价始终是可协商的建议，研究员未以公开相要挟，并非“强买强卖”。

研究员自始按负责任披露流程行事，并在首次会议即完整说明该流程：通过线上会议、脱敏报告、PPT、PoC 演示视频与书面通知向项目方告知风险，给出超过 90 天修复窗口，产品负责人当场确认；并明确——窗口期届满后，无论漏洞是否修复，都将依惯例公开脱敏研究成果。该公开计划自首次沟通即已确定，是负责任披露流程的既定环节。

后续，这批系统级研究成果被项目方按常规社区奖励口径处理、未获任何单独评估或专项赏金；在研究员发出最终通知后，项目方未就诉求继续沟通，转以法务立场回复。截至公开时，多数核心漏洞仍未修复。

研究员依据上述既定的负责任披露计划，公开这份完整脱敏、零可复用武器的研究记录。

## 用户建议

> 以下建议基于客观事实：截至 2026 年 6 月复测，本次整理的 49 个问题中仅 11 个完全修复、9 个部分修复、29 个仍未修复（含多个核心 Critical），且本文公开时未见 Donut 发布第三方安全审计与修复复测报告。下列均为通用的加密资产安全防护措施，适用于任何处于早期阶段、涉及托管钱包、自动签名与自动化执行的产品，并非针对特定主体；是否采取请自行判断。

如果你正在使用 Donut Browser、Donut 钱包、D0 或 Donut AI Agent 相关功能，可优先考虑以下事项。

1. 避免在 Donut 相关钱包中长期存放大额资产，按需充值、用完即提。
2. 定期检查并撤销不必要的 token allowance、合约授权和自动化交易授权。
3. 持续监控链上交易记录，发现异常立即转移资产并保留交易哈希、截图与时间线。
4. 不要在 D0 / AI Agent 环境中处理私钥、助记词、交易所 API Key、内部文档、商业机密或不可公开的个人资料。
5. 如果你使用过自动交易、AI Agent 执行、钱包托管或策略执行功能，可重新评估该账户是否仍适合作为主钱包或高价值账户。
6. 在 Donut 公布第三方安全审计与修复复测结果前，宜按早期、高风险的实验性产品对待，自行控制风险敞口。

## 披露范围与测试边界

本次披露基于授权测试、后续复测、公开沟通和安全研究记录整理而成。为避免误解，披露范围和测试边界如下。

1. 授权测试边界：Donut Browser 阶段发生在 Donut 官方 Bug Hunt 活动及后续线上沟通框架内。相关风险已在会议和书面沟通中告知 Donut。
2. 非破坏性验证边界：研究过程中没有转移任何第三方用户资金，也没有破坏第三方用户数据或服务可用性。
3. 资金验证边界：链上交易验证使用研究员自有测试账户完成，用于证明交易构建与执行链路存在服务端权限边界缺陷。
4. 第三方拦截边界：在跨用户钱包归属校验测试中，研究员使用自己的账号构造他人钱包交易请求，Donut 后端 / API 层仍会构建、提交并推进请求，最终由 Turnkey 返回 AUTH001 拦截资金转出。
5. 租户与容器边界：D0 阶段的命令执行和数据读取验证限定在研究员自有租户 / Pod 内，未在未授权情况下横向攻击其他用户 Pod；D0 阶段的具体发现未单独预先提交，与 Donut Browser 一并纳入本次公开披露。
6. 敏感信息处理边界：披露内容仅包含与事实证明有关的脱敏证据、验证结论和公开依据，不包含可直接复用的访问凭据、私钥或攻击脚本。
7. 披露目的边界：本文用于记录安全研究事实、用户风险、沟通过程和公开依据，不构成攻击指引、投资建议或法律意见。

<a id="project-background"></a>

## 项目背景

Donut 是近一年加密赛道最受瞩目的 AI × Crypto 新锐之一，自称“全球首个面向交易者的智能体（agentic）加密浏览器”。开发商 Donut Labs（总部纽约）成立约**六个月内**即完成两轮、合计 **2200 万美元**融资（2025-05 Pre-seed 700 万 + 2025-11 Seed 1500 万），投资阵容堪称豪华——**红杉中国（HongShan）、BITKRAFT、Hack VC、Makers Fund、Sky9 Capital、Matrix Partners、Altos Ventures** 等顶级机构，外加 Solana、Sui、Monad 等公链与 Jupiter、Drift、Manifold Trading 核心团队站台。据 Donut 官方融资公告，公司**首季即吸引 16 万+ 候补用户**，市场热度极高。

产品上，Donut 以“AI 原生加密浏览器”为愿景：Donut Browser 是浏览器 + 钱包入口，D0 是面向用户的 AI Agent 平台，**核心卖点是让自主智能体直接接管钱包、签名与链上资金执行——CEO（前字节跳动）的原话是“在你睡觉时也能为你交易”**。这也意味着远高于普通应用的安全门槛。正是在这样一个资金充裕、背景顶级、以“AI 自动管钱”为核心卖点的项目上，本次研究于其 Donut Browser 与 D0 两套系统中共发现 49 个安全问题（含自动签名盗币链与自有 Pod RCE）。

<a id="communication-process"></a>

## 沟通过程

本次披露不是突然公开，也不是跳过沟通流程后的单方面曝光。Donut 在漏洞性质、攻击链严重性、修复窗口和披露时间点上均有明确知情机会。

2026 年 3 月 11 日前，我已告知 Donut：发现可能影响用户资金安全的 Critical 级别漏洞，需要与管理层和技术负责人线上沟通。随后，Donut 拉入技术负责人并要求我会后分享脱敏报告。

2026 年 3 月 11 日线上会议中，我演示了 Donut Browser 资金安全攻击链，并说明该问题不是单点参数绕过，而是交易链路、钱包归属校验和信任模型层面的系统性风险。会议中也说明了披露处理方式：先给出沟通与修复窗口，再根据修复情况协调披露；窗口期结束后保留独立公开披露权利。

会后，我向 Donut 提交了脱敏报告、PPT 和 PoC 演示视频。2026 年 3 月 18 日，我再次以书面方式确认披露时间线：基于 2026 年 3 月 11 日会议后的 90 天窗口，计划于 2026 年 6 月 9 日公开披露。

在赏金和后续处理方面：项目方明确表示有 bug bounty 预算，但将本次系统级安全研究纳入常规社区奖励口径处理。两轮 Bug Hunt 活动各发放 100 USDC（合计 200 USDC，分别于 2026 年 2 月 22 日、3 月 28 日到账），是研究者参赛、按活动规则赢得的常规活动奖金，对应的是社区活动级别的常规漏洞（含第二轮独立提交的 13 个）；本次系统级研究（49 漏洞 / 攻击链 / RCE）则未获任何单独或专项赏金。值得注意的是，项目方产品负责人在第二次会议中评价研究者的第一版报告“比花了很多钱找的那个专业团队给的报告，有些角度是比较深的”（见 [第二次会议](./evidence/meetings.md#m2-audit)）——在项目方自己如此评价的同时，该系统级研究仍被按常规社区奖励口径处理。研究者早在 2026-03-18 即书面要求将二者分开评估、反对把系统级研究并入常规社区奖励口径。薪资 / 全职岗位方面，项目方在第三次会议中明确表示“没有这么高的岗位”——这属于明确的“给不到”，与赏金问题应分开看，不可混为一谈。

## Donut Browser 系统

Donut Browser 系统的核心风险集中在“后端替用户构建并推进交易”这条链路上。该系统试图让浏览器、AI 或后端服务帮助用户完成链上交易，但测试中发现，后端没有建立足够严格的用户隔离、钱包归属校验和二次授权边界。

简化链路如下：

```text
用户浏览器
  -> Donut Browser 前端
  -> Donut 后端 API
       -> action-mcp/query        构建交易
       -> action-mcp/execution    执行交易 / 调用签名服务
       -> wallet / portfolio      钱包与资产接口
       -> Turnkey                 第三方密钥托管与策略拦截
```

核心结论分为两部分：第一，1-click 静默交易链已在自有钱包 / 自有资金范围内验证成功，说明用户一次访问或点击即可在未进行任何授权或签名确认的情况下，绕过签名授权流程，由 Donut 后端在无确认下自动签名并广播、将资金转出。第二，跨用户钱包归属校验链显示，研究员使用自己的账号构造他人钱包交易请求时，Donut 后端仍会构建、提交并推进请求，直到 Turnkey 返回 AUTH001 拦截资金转出。

一个典型风险场景如下：

**场景 A：1-click 静默交易链（自有钱包 / 自有资金验证）**

1. 攻击者通过钓鱼入口、恶意页面或可携带凭证的浏览器上下文诱导用户触发一次访问或点击。
2. 浏览器在用户未进行任何授权或签名确认的情况下，携带会话向 Donut 交易接口发起请求。
3. Donut 后端绕过应有的签名授权过程，继续构建并执行链上交易流程。
4. 该链路已在研究员自有钱包 / 自有资金范围内验证成功，证明 1-click 静默交易链成立。

**场景 B：跨用户钱包归属校验链（AUTH001 拦截）**

1. 研究员使用自己的 Donut 账号和会话发起测试。
2. 请求中引用非当前账号所属的钱包，构造从他人钱包发起的交易请求。
3. Donut 后端 / API 层仍然构建、提交并推进该交易请求。
4. 请求进入 Turnkey 策略判断阶段。
5. Turnkey 返回 AUTH001，识别到权限与钱包所属用户不匹配，并拦截最终资金转出。

跨用户钱包归属校验链的严重性在于：Donut 自身没有在 API 层先阻断不属于当前用户的钱包交易请求。对于处理用户钱包和交易的产品，安全边界应当建立在 Donut 自身后端权限模型中，而不是依赖第三方签名策略在最后一刻兜底。

研究中还验证了任意钱包资产、持仓、交易历史查询等 IDOR 风险，以及 CORS、MCP 工具层、额度绕过、AI Agent 配置泄露等组合问题。这些问题可以降低攻击成本，并扩大攻击链可达范围。

<a id="d0-system"></a>

## D0 系统

D0 是 Donut 对外提供的 AI Agent 平台。每个用户会被分配一个独立运行环境 / Pod，前端通过 WebSocket 与对应 Gateway 通信。研究过程中，我通过 D0 漏洞获取到控制面访问材料，并沿着“前端环境接口 -> 控制面连接 -> 配置读取 / 修改 -> 文件写入 -> beforeRun / heartbeat 执行机制 -> 自有 Pod RCE -> Session Token、环境变量和内部服务地址读取”的路径完成验证。访问对应控制面时，页面明确显示 OpenClaw logo，文档入口也指向 OpenClaw 官方文档；这一点用于确认底层控制面来源，漏洞讨论重点仍是 D0 暴露出的控制面权限边界和 Agent 运行环境隔离问题。

D0 系统的核心风险在于：前端环境接口会返回连接底层控制面所需的关键信息，包括公网入口、端口和 Gateway Token。攻击者只要拥有自己的正常 D0 登录态，就可以获取控制面连接信息，并进一步进入配置读取、配置修改、文件写入和执行机制组合链。

简化链路如下：

```text
用户登录 D0
  -> 调用环境接口获取 Gateway 连接信息
  -> 连接底层 OpenClaw WebSocket Gateway
  -> 以高权限角色完成认证
  -> 读取 / 修改配置
  -> 写入文件或修改 beforeRun
  -> 等待 heartbeat 或触发执行路径
  -> 在对应 Pod 内执行命令
  -> 读取 Session Token / ENV / 内部服务地址等信息
```

这条链路的关键问题不是 AI 模型是否“拒绝危险请求”，而是 Agent 运行环境本身存在可被控制面能力影响的执行路径。beforeRun、heartbeat、文件写入和配置修改属于运行时执行机制，不应被前端可获得的普通会话材料轻易触达。

当前验证边界为研究员自有租户 / Pod。研究已证明从控制面连接到命令执行、环境变量读取、Session Token 读取和内部服务地址提取的完整路径存在；未对其他用户 Pod 进行未授权横向攻击。

## 漏洞总览

本节按单个漏洞列出本次披露整理的安全问题。漏洞条目保留编号、名称、影响面、严重性和证据状态，漏洞等级按对资金安全、账户权限、控制面能力、执行环境控制权和攻击链可达性的影响划分。

- **Critical**：可能直接导致资金操作、交易执行、控制面接管、远程代码执行或高权限能力失控，影响用户资产或运行环境控制权。
- **High**：可能造成跨用户越权、敏感凭据暴露、订单或策略被影响、关键配置泄露，或作为 Critical 攻击链的重要组成部分。
- **Medium**：会扩大攻击面、泄露系统或业务信息、降低攻击成本，常用于枚举、探测、资源滥用或后续攻击链准备。
- **Low**：主要体现为安全加固不足、浏览器侧防护缺失、调试信息或生产指纹暴露，单独利用影响有限，但会辅助攻击者建模。

## 值得关注的问题

以下问题来自本文披露的攻击链、沟通记录、漏洞响应过程，以及 Donut 对外安全叙事与实际安全表现之间的落差。

**Q1：Donut 直接经手用户真实资金、可发起并执行不可逆的链上交易。对这样一个产品，用户授权、钱包归属校验、控制面与运行环境隔离等基础安全本应是不可让渡的底线——为什么 Donut Browser 与 D0 仍会在这些基础环节上失守？**

**Q2：Donut 官方文档向用户保证“所有交易都需经你批准（All transactions require your approval）”（见 [官方文档 · Wallet & Security](https://docs.donutbrowser.ai/reference/faq)）。但已验证的 1-click 静默交易链表明：存在可绕过这一批准、在用户未做任何授权或签名确认下即可直接发起并执行交易的路径（自有账户已验证，[11 笔链上交易为证](./evidence/chain.md)）。当一项面向用户的核心安全保证可被绕过，它在实际中还能在多大程度上被依赖？**

**Q3：D0 已验证从普通登录态获取控制面连接材料，并进一步组合到配置读取、配置修改、文件写入、自有 Pod RCE 和敏感会话材料读取（[详见 D0 证据](./evidence/evidence.md#d0sec)）。这样的控制面暴露，是否符合 AI Agent 产品应有的安全边界？**

**Q4：在研究员提交重大安全研究成果、参与会议沟通、提供脱敏报告、PPT 和 PoC 演示视频，并明确给出负责任披露窗口后，项目方仍将 Critical 级漏洞降级为常规社区活动口径处理（项目方原话“是常规社区奖励”见 [Telegram · 2026-03-18](./evidence/telegram.md)，处理经过另见[第三次会议](./evidence/meetings.md)），是否给予了与风险等级相称的重视与响应？**

**Q5：如果独立安全研究员在负责任披露框架下提交重大漏洞成果，最终却被降级为普通社区活动处理，这是否会让更多研究员对合规披露失去信心？**

**Q6：安全研究的投入与回报，在攻击者与白帽之间高度不对称：攻击者只需找到一条可利用路径，无需顾及授权、合规或是否被认可；白帽却要在授权边界、负责任披露与长期沟通上付出大量成本，最终可能连一句公开致谢都没有；甚至在合规框架内正当争取合理回报，也可能被重新定性为“施压”或“敲诈”、进而面临法律威胁。当合规披露长期得不到与风险相称的回应、研究者因此流失，会不会反而削弱整个行业所依赖的协调披露机制——而真正的攻击者，从不需要这些前提？**

**Q7：Donut 在官方介绍核心产品 D0 时强调“金融操作不可逆，对安全的要求从根本上就不同（the bar for safety is fundamentally different）”（[官方 X @DonutAI](https://x.com/DonutAI/status/2052440302003081555)）。但 D0 恰恰基于一款上线时间很短的开源项目二次开发，本次研究并在其上取得了 RCE 与控制面接管——这与其安全定位、融资规模和市场期待是否相称？相应的潜在安全风险，是否得到了与之匹配的安全投入？**

<a id="security-governance"></a>
## 安全治理观察

这次事件不只是若干接口漏洞的集合，更暴露了 AI + Crypto 产品在安全治理上的几个共性问题。

第一，AI Agent 一旦接入钱包、交易、签名、策略执行和链上资产，就不再是普通聊天机器人，而是带有金融执行能力的自动化系统。这样的系统需要强制的后端权限模型、资产归属校验、最小权限、签名确认、审计日志和异常拦截。

第二，第三方安全策略不能替代项目自身的权限模型。Turnkey 返回 AUTH001 并拦截高风险交易，说明第三方服务在最后阶段发挥了拦截作用。Donut 自身后端仍暴露出用户与钱包、订单、交易之间的绑定校验问题。

第三，Bug Hunt 或社区反馈机制不能替代正式漏洞响应流程。当项目公开鼓励社区测试时，也应准备好处理真正的 Critical 级安全事件，包括分级评估、修复对接、披露协调、复测反馈和合理奖励。

第四，AI Agent 运行环境的控制面安全应被视为核心资产边界。前端会话、Gateway Token、配置修改、文件写入、beforeRun 和 heartbeat 之间如果缺少强隔离，就可能让攻击者绕过模型层安全审查，直接触达执行层。

第五，开源依赖和二次开发本身不是问题。问题在于，一个对外披露两轮合计融资 2200 万美元、并将 D0 作为核心 AI Agent 产品呈现的公司，其核心控制面在实际访问中明确显示 OpenClaw logo，文档入口也指向 OpenClaw 官方文档。这样的事实应当被公开讨论：D0 对外包装、自研叙事、关键开源依赖透明度、二次开发安全审查和实际安全工程能力之间，存在明显落差。真正值得关注的不是使用开源，而是在高融资叙事下没有充分说明核心产品对 OpenClaw 的依赖，也没有体现出与资金规模相匹配的安全工程和漏洞响应能力。


第六，安全研究的认可机制本身也是安全治理的一部分。成熟的负责任披露体系通常以专项赏金、公开致谢（官网致谢、安全名人堂、披露致谢）等方式回馈研究者，这既是对研究价值的承认，也是吸引持续、合规安全研究的基础。本次系统级研究（49 漏洞 / 攻击链 / RCE）既未获任何单独评估或专项赏金，也未在任何公开渠道获得正式认可（仅 Discord 社区有非正式感谢）。需要说明的是，安全研究无偿提交在行业中并不少见，研究者也从未将赏金作为公开披露的前提；这里的观察点不在于“是否付费”，而在于：项目方自身明确表示设有 bug bounty 预算，却仍将此类系统级研究按常规社区奖励口径处理。如何对待重大安全研究成果，本身就反映了一个产品的安全治理成熟度。

本文记录这些问题的目的，是让用户、媒体、投资人和开发团队更清楚地看到：AI + Crypto 产品的安全能力不应只体现在叙事和界面上，而应体现在后端权限模型、运行环境隔离和漏洞响应机制中。

<a id="legal-basis"></a>

## 披露依据

本节不是法律意见，而是说明本次公开披露所依据的事实基础和安全行业通行流程。

本次披露符合负责任披露 / 协调披露的基本逻辑：研究员在发现高风险问题后，先通过线上会议和书面材料向项目方说明漏洞性质、影响范围和修复窗口；项目方在合理时间内拥有知情、评估、修复和沟通机会；窗口期届满后，研究员可以在不公开可直接滥用细节的前提下，向用户和公众披露风险。

本次披露的事实基础包括：

1. Donut Browser 阶段测试发生在 Donut 官方 Bug Hunt 活动及后续线上沟通框架内。
