<a id="summary"></a>

## 摘要

这是一份关于 Donut AI 相关产品安全风险的公开披露。披露对象主要包括 Donut Browser 浏览器钱包系统，以及 D0 / OpenClaw AI Agent 系统。

简单说，这些问题说明：当 AI 产品开始接入钱包、交易、自动化执行和用户账户环境时，它已经不再只是一个聊天工具，而是在接近金融执行系统。这样的系统如果缺少严格的权限边界、钱包归属校验和运行环境隔离，就可能把普通用户暴露在资金、隐私和账户控制风险之下。

本次研究共整理 49 个安全问题：Donut Browser 35 个，D0 / OpenClaw 14 个。Donut Browser 部分的核心风险是后端交易构建与执行链路缺少足够的用户隔离；D0 / OpenClaw 部分的核心风险是用户 AI Agent 运行环境、控制面能力和敏感会话材料之间的边界过弱。

在 Donut Browser 测试中，研究验证了跨用户钱包交易请求可以被 Donut 后端继续推进，直到第三方 Turnkey 策略在最后阶段拦截。也就是说，最后阻止第三方资金被转出的关键机制来自外部托管服务，而不是 Donut 自身后端权限模型。

在 D0 / OpenClaw 测试中，研究验证了从前端环境信息、控制面访问到容器内命令执行和敏感数据读取的完整链路。相关验证限定在研究员自有租户 / Pod 范围内，没有对其他用户环境进行横向攻击。

本文面向普通用户、媒体、安全从业者和行业观察者。文中会保留必要的事实、时间线、截图占位、技术结论和公开依据，但不会公开 live token、JWT、Cookie、私钥、反弹 Shell 参数或一键化 exploit 脚本。

<a id="user-advice"></a>

## 用户建议

如果你正在使用 Donut Browser、Donut 钱包、D0 或 Donut AI Agent 相关功能，建议优先处理以下事项。

1. 不要在 Donut 相关钱包中长期存放大额资产。
2. 检查并撤销不必要的 token allowance、合约授权和自动化交易授权。
3. 监控你的链上交易记录，发现异常后立即转移资产并保留证据。
4. 不要在 D0 / AI Agent 环境中处理私钥、助记词、交易所 API Key、内部文档、商业机密或不可公开的个人资料。
5. 如果你使用过自动交易、AI Agent 执行、钱包托管或策略执行功能，应当重新评估该账户是否仍适合作为主钱包或高价值账户。
6. 在 Donut 公布清晰的安全整改、外部审计和复测报告前，建议将相关产品视为高风险实验性产品。

<a id="project-background"></a>

## 项目背景

Donut 是一个 AI + Crypto 产品生态，产品方向包括浏览器、钱包、链上交易辅助、AI Agent 和自动化执行等能力。其核心叙事是让用户通过更自然的方式接入链上资产、策略执行和交易操作。

Donut Browser 承担浏览器与钱包入口角色，涉及用户钱包、资产查询、交易构建、DeFi 操作、限价单和后端交易执行链路。D0 则是 Donut 生态中的 AI Agent 平台，基于 OpenClaw 运行，用户可以在独立环境中配置和运行 Agent。

这类产品的安全风险不同于普通 Web 应用。普通 Web 应用的权限问题可能导致数据泄露或账户功能滥用；而 AI + Crypto 产品一旦连接钱包、签名服务、自动交易和容器运行环境，权限缺陷就可能直接影响资产安全、会话安全和用户运行环境控制权。

因此，对 Donut 这类系统而言，安全边界至少应包括：后端用户身份与钱包归属绑定、交易执行前的强授权确认、最小权限控制、敏感凭据隔离、Agent 运行环境隔离、控制面能力审计，以及异常操作拦截机制。

本次披露讨论的正是这些边界在实际测试中暴露出的系统性问题。

<a id="scope-boundaries"></a>

## 披露范围与测试边界

本次披露基于授权测试、后续复测、公开沟通和安全研究记录整理而成。为避免误解，披露范围和测试边界如下。

1. 授权测试边界：Donut Browser 阶段发生在 Donut 官方 Bug Hunt 活动及后续线上沟通框架内。相关风险已在会议和书面沟通中告知 Donut。
2. 非破坏性验证边界：研究过程中没有转移任何第三方用户资金，也没有破坏第三方用户数据或服务可用性。
3. 资金验证边界：链上交易验证使用研究员自有测试账户完成，用于证明交易构建与执行链路存在服务端权限边界缺陷。
4. 第三方拦截边界：Donut Browser 跨用户资金攻击链在 Donut 后端 / API 层可继续推进，但实际第三方资金转出被 Turnkey 策略拦截。本文会明确区分 Donut 自身缺陷与第三方安全策略的拦截结果。
5. 租户与容器边界：D0 / OpenClaw 阶段的命令执行和数据读取验证限定在研究员自有租户 / Pod 内。未在未授权情况下横向攻击其他用户 Pod。
6. 敏感信息处理边界：正式公开版本不会包含可直接复用的 live token、JWT、Cookie、私钥、反弹 Shell 参数、一键 exploit 或仍有效的内部服务访问材料。
7. 披露目的边界：本文用于记录安全研究事实、用户风险、沟通过程和公开依据，不构成攻击指引、投资建议或法律意见。

## 沟通过程

本次披露不是突然公开，也不是跳过沟通流程后的单方面曝光。Donut 在漏洞性质、攻击链严重性、修复窗口和披露时间点上均有明确知情机会。

2026 年 3 月 11 日前，我已告知 Donut：发现可能影响用户资金安全的 Critical 级别漏洞，需要与管理层和技术负责人线上沟通。随后，Donut 拉入技术负责人并要求我会后分享脱敏报告。

2026 年 3 月 11 日线上会议中，我演示了 Donut Browser 资金安全攻击链，并说明该问题不是单点参数绕过，而是交易链路、钱包归属校验和信任模型层面的系统性风险。会议中也说明了披露处理方式：先给出沟通与修复窗口，再根据修复情况协调披露；窗口期结束后保留独立公开披露权利。

会后，我向 Donut 提交了脱敏报告、PPT 和 PoC 演示视频。2026 年 3 月 18 日，我再次以书面方式确认披露时间线：基于 2026 年 3 月 11 日会议后的 90 天窗口，计划于 2026 年 6 月 9 日公开披露。

在赏金和后续处理方面，Donut 最终将该事件纳入普通社区奖励路径处理，而不是建立针对系统级资金安全问题的专项评估、修复对接或正式漏洞赏金流程。该处理方式是本文记录的安全治理问题之一。

在沟通过程受阻后，我曾与 Donut D0 平台中的 AI Agent 讨论该事件。该内容不能替代事实证据，但作为旁证，它反映出即使在抽象描述层面，Critical 级资金攻击链、私下报告、修复窗口和公开披露之间的关系，也应被严肃对待。

<a id="donut-browser"></a>

## Donut Browser 系统

Donut Browser 系统的核心风险集中在“后端替用户构建并推进交易”这条链路上。该系统试图让浏览器、AI 或后端服务帮助用户完成链上交易，但测试中发现，后端没有建立足够严格的用户隔离、钱包归属校验和二次授权边界。

简化链路如下：

```text
用户浏览器
  -> Donut Browser 前端
  -> Donut 后端 API
       -> action-mcp/query        构建交易
       -> action-mcp/execution    执行交易 / 调用签名服务
       -> wallet / portfolio      钱包与资产接口
       -> Turnkey                 第三方密钥托管与策略拦截
```

核心结论是：Donut 后端允许攻击者构建并提交不属于当前会话用户的钱包交易，且执行链路会继续向后推进，直到第三方 Turnkey 策略在最后阶段拦截。

一个典型风险场景如下：

1. 攻击者拥有一个普通 Donut 账户。
2. 攻击者通过接口查询或构造目标钱包相关参数。
3. 攻击者提交一笔从目标钱包发起的交易构建请求。
4. Donut 后端没有在 API 层严格验证该钱包是否属于当前会话用户。
5. 后端继续推进交易构建和执行链路。
6. Turnkey 策略在最后阶段拦截非授权资金转出。

这类问题的严重性不取决于“这一次是否刚好被第三方拦住”，而取决于 Donut 自身是否具备最基本的资产归属校验和交易授权边界。对于处理用户钱包和交易的产品，安全边界应当建立在 Donut 自身后端权限模型中，而不是依赖第三方签名策略在最后一刻兜底。

研究中还验证了任意钱包资产、持仓、交易历史查询等 IDOR 风险，以及 CORS、MCP 工具层、额度绕过、AI Agent 配置泄露等组合问题。这些问题可以降低攻击成本，并扩大攻击链可达范围。

<a id="d0-openclaw"></a>

## D0 / OpenClaw 系统

D0 是 Donut 生态中的 AI Agent 平台，基于 OpenClaw 运行。每个用户会被分配一个独立运行环境 / Pod，前端通过 WebSocket 与对应 Gateway 通信。

D0 / OpenClaw 系统的核心风险在于：连接控制面所需的关键信息会被前端环境接口返回，包括公网入口、端口和 Gateway Token。攻击者只要拥有自己的正常 D0 登录态，就可以获取自己的控制面连接信息，并进一步调用 OpenClaw 控制面能力。

简化链路如下：

```text
用户登录 D0
  -> 调用环境接口获取 Gateway 连接信息
  -> 连接 OpenClaw WebSocket Gateway
  -> 以高权限角色完成认证
  -> 读取 / 修改配置
  -> 写入文件或修改 beforeRun
  -> 等待 heartbeat 或触发执行路径
  -> 在对应 Pod 内执行命令
  -> 读取 Session Token / ENV / 内部服务地址等信息
```

这条链路的关键问题不是 AI 模型是否“拒绝危险请求”，而是 Agent 运行环境本身存在可被控制面能力影响的执行路径。beforeRun、heartbeat、文件写入和配置修改属于运行时执行机制，不应被前端可获得的普通会话材料轻易触达。

当前验证边界为研究员自有租户 / Pod。研究已证明从控制面连接到命令执行、环境变量读取、Session Token 读取和内部服务地址提取的完整路径存在；未对其他用户 Pod 进行未授权横向攻击。

正式公开版本会删除仍可能直接复用的 Token、JWT、IP、反弹 Shell 参数和一键化脚本，仅保留脱敏后的请求路径、风险解释、截图占位和证据哈希。

## 漏洞总览

本节列出公开披露版本的漏洞总览。这里保留漏洞名称、影响面和证据状态，不放置可直接复用的 exploit、live token、JWT、Cookie、反弹 Shell 参数或一键攻击脚本。最终发布前会补充 CVSS / CWE 校对结果和 2026 年 6 月初复测状态。

### Donut Browser 漏洞 DB-1 至 DB-35

| 编号 | 漏洞名称 | 严重性 | 类型 | 影响与证据状态 |
|---|---|---|---|---|
| DB-1 | 服务端交易自动签名 / 执行链路缺失强授权边界 | Critical | 资金安全 / 授权缺陷 | 后端可继续推进非当前用户钱包交易请求，链路已通过自有账户链上交易验证，跨用户资金转出由 Turnkey 最后拦截 |
| DB-2 | 跨用户交易构建未绑定当前会话钱包归属 | Critical | IDOR / 业务逻辑 | 交易构建阶段未充分证明 `from_wallet` 属于当前会话用户，可作为资金攻击链入口 |
| DB-3 | 任意钱包资产、持仓、交易历史 IDOR | Critical | 越权访问 / 信息泄露 | 可读取非当前用户钱包的资产组合、成本、持仓和历史信息，已发现高价值钱包样本 |
| DB-4 | CORS 子域通配符与 credentials 组合 | Critical | Web 配置错误 | 恶意子域或被控子域可携带登录态调用敏感 API，放大为远程触发链路 |
| DB-5 | MCP 工具层认证与权限边界不足 | Critical | 认证绕过 / 工具滥用 | MCP 工具可被枚举或直接调用，绕过前端限制并进入工具执行层 |
| DB-6 | Role 参数注入 | High | 输入校验 / 权限提升 | 可伪造 system/assistant 等高权限消息角色，削弱 AI Agent 行为边界 |
| DB-7 | Wallet Service 无认证或弱认证钱包创建 | High | 认证缺失 / 资源滥用 | 可跨 13 条链创建钱包，造成资源消耗、滥用和合规风险 |
| DB-8 | Credits / 使用次数限制绕过 | High | 业务逻辑 | 可绕过 AI Agent 使用额度和订阅限制，降低攻击成本并影响商业模型 |
| DB-9 | AI Agent 配置与 System Prompt 泄露 | High | 信息泄露 / Prompt Injection 原料 | 多个 Agent 的 Prompt、工具配置和调用规则暴露，可用于定向提示词攻击 |
| DB-10 | 跨用户限价单取消 | High | 越权访问 | 订单取消缺少钱包归属校验，可能影响其他用户交易策略和资产操作 |
| DB-11 | DeFi 自动签名交易链路缺少用户确认 | Critical | 资金安全 | DeFi 操作可进入后端签名链路，已通过自有账户交易证明执行路径存在 |
| DB-12 | Admin Plan / 订阅计划信息泄露 | High | 信息泄露 / 业务逻辑 | 管理计划、订阅策略或 taskKey 相关信息暴露，可用于绕过或滥用订阅体系 |
| DB-13 | SQL 查询结构与表结构泄露 | High | 信息泄露 | 错误响应或接口返回暴露 SQL 结构、字段和业务表信息，便于后续攻击建模 |

<a id="security-governance"></a>

## 安全治理观察

这次事件不只是若干接口漏洞的集合，更暴露了 AI + Crypto 产品在安全治理上的几个共性问题。

第一，AI Agent 一旦接入钱包、交易、签名、策略执行和链上资产，就不再是普通聊天机器人，而是带有金融执行能力的自动化系统。这样的系统需要强制的后端权限模型、资产归属校验、最小权限、签名确认、审计日志和异常拦截。

第二，第三方安全策略不能替代项目自身的权限模型。Turnkey 在最后阶段拦截了高风险交易，这是第三方服务发挥作用的结果，但项目方仍应在自身后端完成用户与钱包、订单、交易之间的强绑定校验。

第三，Bug Hunt 或社区反馈机制不能替代正式漏洞响应流程。当项目公开鼓励社区测试时，也应准备好处理真正的 Critical 级安全事件，包括分级评估、修复对接、披露协调、复测反馈和合理奖励。

第四，AI Agent 运行环境的控制面安全应被视为核心资产边界。前端会话、Gateway Token、配置修改、文件写入、beforeRun 和 heartbeat 之间如果缺少强隔离，就可能让攻击者绕过模型层安全审查，直接触达执行层。

本文记录这些问题的目的，是让用户、媒体、投资人和开发团队更清楚地看到：AI + Crypto 产品的安全能力不应只体现在叙事和界面上，而应体现在后端权限模型、运行环境隔离和漏洞响应机制中。

<a id="legal-basis"></a>

## 法律依据

本节不是法律意见，而是说明本次公开披露所依据的事实基础和安全行业通行流程。

本次披露符合负责任披露 / 协调披露的基本逻辑：研究员在发现高风险问题后，先通过线上会议和书面材料向项目方说明漏洞性质、影响范围和修复窗口；项目方在合理时间内拥有知情、评估、修复和沟通机会；窗口期届满后，研究员可以在不公开可直接滥用细节的前提下，向用户和公众披露风险。

本次披露的事实基础包括：

1. Donut Browser 阶段测试发生在 Donut 官方 Bug Hunt 活动及后续线上沟通框架内。
2. 2026 年 3 月 11 日，我已通过线上会议向 Donut 演示并说明核心风险。
3. 会后我向 Donut 提交了脱敏报告、PPT 和 PoC 演示材料。
4. 2026 年 3 月 18 日，我以书面方式明确通知计划于 2026 年 6 月 9 日公开披露。
5. 测试过程中未转移第三方用户资金，D0 / OpenClaw 的命令执行验证限定在研究员自有租户 / Pod 内。
6. 正式公开版本会删除可直接复用的 live token、JWT、Cookie、私钥、反弹 Shell 参数和一键化 exploit。

可参考的行业流程包括：

- [ISO/IEC 29147:2018 - Vulnerability disclosure](https://www.iso.org/standard/72311.html)：描述漏洞披露中供应商接收报告、披露信息和降低用户风险的要求与建议。
- [ISO/IEC 30111:2019 - Vulnerability handling processes](https://www.iso.org/standard/69725.html)：描述漏洞处理流程。
- [CERT/CC Coordinated Vulnerability Disclosure Guide](https://certcc.github.io/CERT-Guide-to-CVD/)：协调漏洞披露的实践指南。
- [CERT-EU Coordinated Vulnerability Disclosure Policy](https://www.cert.europa.eu/coordinated-vulnerability-disclosure-policy)：公开披露窗口和协调披露实践参考。

基于上述事实，本文的公开目的在于告知用户风险、记录安全研究过程、推动项目方和行业重视 AI + Crypto 产品的系统性安全边界。
