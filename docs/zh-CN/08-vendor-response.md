## 项目方回应

### 从”会给答复”到”社区奖励”的含糊与推诿

在 2026 年 3 月 11 日线上会议中，我在演示完攻击链后，明确提出了漏洞赏金和后续合作意向。Donut 团队表示**会给我一个明确答复**。

之后的沟通中，Donut 使用了 **”GTM Bug Bounty”** 这个说法，并提到我之前参加第一次 Bug Hunt 时拿过 bounty，这次也会发放并给予 special shout out。听起来像是一个有预算、有评估机制的正式流程。

然而，在我一再追问具体金额和评估标准后，Donut 的真实态度才逐渐暴露：所谓 “GTM Bug Bounty” 就是**常规社区奖励**，每人约 $100。没有专项评估，没有按漏洞严重性定价，没有参考任何行业标准。

**从一开始的含糊其辞，到最后被追问才承认是社区奖励** — 这个过程本身就说明 Donut 清楚知道 $100 对不起这些漏洞的价值，但选择用模糊话术拖延，直到研究员主动要求澄清。

### 赏金博弈

运营私聊询问我对安全赏金的预期。我的回复很明确：金额不适合由研究员自行提出，因为这可能产生”威胁倾向”的误解；我建议 Donut 团队根据漏洞影响进行内部评估，并参考行业数据给出报价。

Donut 的回应是：额外奖金也考虑了我此前提出的”安全研究原则和威胁倾向”，所以他们决定以”中立视角”来定义本次 bounty。

翻译一下：**我为了避免被误解为威胁而没有自行报价，反而被 Donut 用作不做专项赏金评估的理由。**

### 关于”感谢”

最终，Donut 对这次安全研究的全部公开感谢，是在 **Discord 频道中发了一条消息**。

没有官网致谢页面。没有官方推特声明。没有安全公告。没有 Hall of Fame。

一个融资 2200 万美元的项目，面对 49 个漏洞（含多个 CVSS 10.0）、11 笔链上交易证明、RCE 和反弹 Shell，给出的回应是：Discord 里一条感谢消息 + $100。

### 关于 $100

这与我在 2026 年 2 月第一次 Bug Hunt 中获得的奖金完全相同——尽管第二次的发现从规模、影响和验证深度上已经完全不是同一个量级：

| | 第一次 Bug Hunt（2月） | 第二次 Bug Hunt 2.0（3月） |
|---|---|---|
| 漏洞数量 | 8 个 | 49 个 |
| 严重性 | 功能性 Bug + 安全问题 | 多个 CVSS 10.0 Critical |
| 验证深度 | 提交报告 | 11 笔链上交易 + RCE + 反弹 Shell |
| 交付物 | Bug 报告 | 会议演示 + PPT + PoC 视频 + 脱敏报告 |
| 获得赏金 | $100 | $100 |

一个融资 2200 万美元、处理用户钱包和交易的 AI + Crypto 项目，面对系统级资金安全攻击链，如果两次都只能给出 $100 社区奖励——那么这个处理方式本身就是本文值得公开记录的一部分。

### 为什么我认为这不是普通社区反馈

这次提交不是”页面报错””功能异常”或普通参数问题，而是：

- 交易链路和信任模型层面的系统级风险；
- 可以推动服务端交易构建和执行链路，攻击者无需受害者任何参与；
- 涉及用户钱包、链上资金、AI Agent、后端控制面和容器运行环境；
- 已通过链上交易、会议演示、PoC 视频和复测记录证明影响；
- 包含多个 CVSS 10.0 的 Critical 级别问题；
- 影响 1,375+ 活跃钱包和 16 万+ waitlist 用户。

如果这类问题最终仍被纳入普通社区预算处理，那么所谓 Bug Bounty 对安全研究员而言就失去了基本可信度。
